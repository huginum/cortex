= Local Setup

Required tools:

* Rust and Cargo
* Node.js and npm
* Tauri CLI: `cargo install tauri-cli --locked`
* Zig 0.15.x for building libghostty-vt WebAssembly
* Antora CLI and site generator: `npm i -g @antora/cli@3.1 @antora/site-generator@3.1`
* macOS Command Line Tools
* libkrun (for microVM sandboxes) — see <<sandboxes>>

Useful commands:

* `npm install`
* `npm run ghostty:wasm`
* `npm run ghostty:verify`
* `npm run tauri:dev`
* `npm run docs:build`

Ghostty currently requires Zig 0.15.x for the libghostty-vt build. If Homebrew installs it as keg-only, the build script looks for `/opt/homebrew/opt/zig@0.15/bin/zig` automatically.

[#sandboxes]
== MicroVM sandboxes (libkrun)

Cortex runs sandbox terminals as microVMs by linking https://github.com/containers/libkrun[libkrun] directly. libkrun is a *build-time dependency*: the backend calls its C API, so it must be installed to compile Cortex.

Sandboxes run on Apple Silicon macOS (Hypervisor.framework) and on Linux (KVM). They are not available on Intel Macs or Windows. Host shells are unaffected and need none of this.

=== Install libkrun

macOS (Apple Silicon):

[source,sh]
----
brew tap libkrun/krun
brew trust libkrun/krun       # the tap ships formulae that run code at install time
brew install libkrun          # also installs libkrunfw (the guest kernel)
----

libkrun is keg-only; the build script resolves its location with `brew --prefix libkrun`. To use a non-Homebrew build, set `LIBKRUN_LIB_DIR` to the directory containing `libkrun.dylib`.

Linux:

* Install `libkrun` and `libkrunfw` (from your distribution or `make install`); set `LIBKRUN_LIB_DIR` if they are not on the default library path.
* The user running Cortex needs access to `/dev/kvm` (add your user to the `kvm` group).

=== macOS code signing (required for HVF)

Hypervisor.framework refuses to start in a process that is not signed with the `com.apple.security.hypervisor` entitlement. The entitlements live in `src-tauri/Entitlements.plist` (hypervisor access, plus disabled library validation so the Homebrew libkrun/libkrunfw dylibs load under the hardened runtime).

The GUI process itself does not need the entitlement — only the on-disk binary does, because the sandbox helper is a fresh `exec` of that binary (which the kernel evaluates against the file's signature). So sign the binary *while the app is running*, with no relaunch:

[source,sh]
----
npm run tauri:dev          # terminal 1: builds and launches; leave it running
npm run sandbox:sign-dev   # terminal 2: AFTER the window is up, sign target/debug/cortex
----

Then start a sandbox — the running app execs the freshly signed binary for the helper. Do **not** sign before `tauri:dev` or relaunch with it afterward: `tauri dev` recompiles and replaces (unsigns) the binary. Re-sign after any rebuild.

To avoid the two-terminal dance, `npm run dev:signed` builds, signs, and runs the signed binary against the Vite dev server in one command.

Release builds (`npm run tauri:build`) pick up `Entitlements.plist` from `tauri.conf.json`; set your Developer ID signing identity in the environment so the bundle is signed and notarizable.

=== Running containers

Open the sandbox menu in the project toolbar (the cube icon), type an image reference (e.g. `ubuntu:24.04` or `alpine:latest`) and press Run, or pick a previously fetched image. Cortex pulls the image in pure Rust (no Docker/skopeo/buildah), unpacks its layers into a cache, and boots a microVM from it into a terminal pane.

Images are pulled for the host CPU architecture's Linux variant and cached under the app data dir:

[source,sh]
----
# macOS
~/Library/Application Support/com.huginum.cortex/images/<ref>/rootfs/
# Linux
~/.local/share/com.huginum.cortex/images/<ref>/rootfs/
----

v1 pulls public images anonymously; private-registry authentication is not yet supported.

=== Self-contained Linux build (AppImage)

The Linux AppImage embeds libkrun and libkrunfw, so it runs sandboxes without a separate VMM install (the host still needs `/dev/kvm`).

Build it on an *ARM Linux host* — arm64 AppImages can only be built on ARM hardware. The build host needs the toolchain (Rust, Node, WebKitGTK 4.1, and libkrun for linking; set `LIBKRUN_LIB_DIR` if libkrun is not on the default path):

[source,sh]
----
sh scripts/build-appimage.sh
----

The script stages `libkrun.so.1` and `libkrunfw.so.5` into `src-tauri/appimage-libs/` (referenced by `tauri.conf.json`), then builds the AppImage. If your installed sonames differ (e.g. `libkrunfw.so.6`), update the `bundle.linux.appimage.files` map in `tauri.conf.json` to match. At runtime the sandbox helper sets `LD_LIBRARY_PATH` to the bundled library directory.

Sandboxes on Linux require `/dev/kvm`: ensure KVM is enabled and your user is in the `kvm` group.
