= 0003: Projects Are Git Repositories

Status: Accepted

== Context

The bootstrap app opened directly into a single local terminal that owned the
window, and quit when that shell exited. There was no way to organize work,
return to a previous setup, or run more than one terminal. We want the unit of
work to be a project the user opens — the way a DAW or video editor opens a
project — rather than a lone shell.

== Decision

A project is a git repository. Its repository root anchors the project, and
terminal working directories are stored relative to that root.

* *Launch shows a project picker*, not a terminal. The picker lists recently
  opened projects, or offers to add one when none exist.
* *Adding a project* supports opening an existing repository, opening a folder
  and running `git init`, or cloning a remote.
* *A project holds a tree of split panes* (horizontal/vertical), each pane a
  terminal session.
* *Local-only persistence*: both the recent-projects list and the per-project
  layout (split tree plus per-pane relative working directory) live in Cortex's
  application config directory, keyed by the canonical repository root path.
  Cortex never reads or writes inside the repository working tree.
* *Session lifecycle is decoupled from app lifecycle*: a pane's shell exiting
  closes that pane only; the app quits only when the user quits it.

[NOTE]
====
The layout-storage decision was revised by change `local-layout-storage`.
Layout originally lived in `<repo>/.cortex/layout.json`; reading and writing a
repo-controlled path proved to be a persistent source of filesystem
vulnerabilities (symlink overwrite, hard-link truncation, FIFO hangs,
directory-inode TOCTOU). Storage moved to the application config directory,
which the repository cannot influence. The trade-off: a layout can no longer be
committed to share a default workspace across clones.
====

== Consequences

Reopening a project restores its layout and spawns *fresh* shells in the saved
working directories. A saved layout describes panes and directories only — never
live process state or scrollback — because a shell is a live process that cannot
be resurrected across restarts.

Because layout lives in the application config directory and not in the working
tree, opening a project never adds anything to `git status`, and the repository
is never touched for layout storage. Layouts are therefore machine-local: they
do not travel to other clones. Both the recent-projects list and the layout
store can reference repositories that were moved or deleted; the picker detects
stale recent entries and offers to remove them, and a moved repository simply
opens with a fresh, empty layout.

Git operations (validate, init, clone, branch) shell out to the system `git`
binary rather than adding a Rust git dependency, keeping the dependency surface
minimal for the simple operations v1 needs.

== Alternatives Considered

* *Store layout in `<repo>/.cortex/layout.json`* (the original decision):
  superseded. Keeping layout in the working copy allowed committing it to share
  a workspace, but reading/writing a repo-controlled path was a recurring
  security liability; see the note above.
* *Auto-manage git ignores for an in-repo layout* (append `.cortex/` to
  `.gitignore` or `.git/info/exclude`): moot now that nothing is written in the
  repository. When layout was in-repo, both were rejected — editing tracked
  `.gitignore` mutates a shared file, and silently writing `.git/info/exclude`
  pre-decides an ignore-vs-commit question that belonged to the user.
* *Restore scrollback as replayed text*: rejected for v1; it would imply a live
  session that no longer exists.
* *Use a Rust git library (e.g. gix)*: deferred; a heavier dependency than the
  init/clone/status operations warrant today.
